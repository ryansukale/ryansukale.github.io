Wanderings and learnings : April 2014

Apr 29

  1. After getting a feeling that I was simply overloading the simplicity of bootcamp by adding marionettejs to it, I decided to shift the marionettejs branch to a whole new repository. Marionettejs is really good for building large scale SPA and if you are simply creating a moderately sized application, you should not have to use all of the default setup of marionette objects. Hence is born - bootnette. A spin off from bootcamp which draws upon bootcamp and extends it for integration with marionettejs. I am not sure what this project will finally evolve into, but I hope it at least turns into a good chunk of reusable code and patterns that can be used in many marionettjs/bootstrap based SPA's.
  2. The App in marionettjs is analogous to a globa object. Its all knowing, and is the place where you need to identify "regions". These regions are then passed around to controllers who are aware of what to display in those regions. However, as per Brian Mann's brilliant video, it seems that the controller is meant to live within an app module and is the place where you would end up writing most of your code.
  3. Integrated handlebars into bootnette. Best practices guide that you should not emebed script like code in your markup. That makes underscore templating a big no no.

Apr 27

  1. Continued work on xnode. Realized that the new version of expressjs does quite a few things differently. E.g. as per this blog post, you are not supposed to use bodyParser middleware directly and suggests to use json and urlEncoded instead. These were old recommendations. The new bodyparser uses just json and urlencoded and therfore it is safe to use bodyParser again.
  2. This link has a neat table of all the middleware that you will now need to explicitly require for your application since express no longer depends on connect.
  3. Express router seems to be much more neat.
  4. When you test your REST API in Postman, just remember to use the xxx-form-urlencoded instead of form-data, or your POST data will not show up in req.body in express.
  5. Expressjs has a pretty cool History.md file. I added the same to xnode.
  6. An express router can take multiple callbacks as arguments before the actual handler function is executed. This is quite similar to middlware execution, except for the fact that it can be made more route specific. Within each callback, invoking next() calls the next callback, and calling next('router') skips the subsequent callbacks and directy calls the final handler.

Apr 24

  1. Building bootcamp is gradually becoming more and more interesting. After reading a couple of posts here, here and here, I figured a way to setup Marionettejs modules such that they can be initialized in an requirejs style.
  2. You can get the current module's name from within the module in requirejs by module.id
  3. Faced a cool error Module name has not been loaded yet for context:_. This happens when you require('moduleName') a module that was not already loaded as part of the dependencies. As a solution, you use the require([],function(){}) syntax instead. Here's an SO question that talks more about this problem.
  4. You dont need to require 'require' as a module. If you are using requirejs, you already have access to require from inside your module functions. When in doubt you can stil do it this way.
  5. Marionette application does not have a before:start event. Only modules have that. Damn it, i wasted so much time wondering what I was doing wrong.
  6. Was setting up the xnode project. And thats when I realized that many of the utilities like methodOverride, json etc now need to be imported as independent packages.

Apr 21

  1. Read up a little bit on git rebase from here
  2. When you generate ssh keys, if you dont use sudo, the keys are created for the current user. If you use sudo, the keys are created for the root user.
  3. This is an important thing to know because when you do git clone via ssh, you need to be using the correct user. If you get a permission denied error on git clone, thats probably the reason why.ref
  4. When you are generating ssh keys using github, and you run the command sudo ssh-add ~/.ssh/id_rsa and you get the error - 'Could not open a connection to your authentication agent.', you need to run the command eval `ssh-agent -s` ref. You might also need to run eval "$(ssh-agent)" and maybe this too in case ssh-add is not recognized.
